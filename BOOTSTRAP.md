# tkr — Agent Project Bootstrap Guide

> **Audience:** AI coding agent (Claude Code, Cursor, Copilot Workspace, etc.)
> **Goal:** Create the complete project skeleton so a human developer can open it and immediately begin implementing features from `TASKS.md` without any setup friction.
> **When you are done**, the human should be able to run `make build` and get a clean compile, `make test` and get zero failures, and open `TASKS.md` to start on M0.

---

## 0. Before You Start

Read all four documents in order before creating a single file:

1. `.agent/FUNCTIONAL_SPEC.md` — authoritative source for all behaviour, types, interfaces
2. `.agent/API_PROVIDERS.md` — provider details you will wire into config templates
3. `.agent/AI_AGENT_GUIDE.md` — coding rules, approved deps, interface contracts
4. `.agent/skills/tkr-index/SKILL.md` — skill map; consult it whenever you are unsure which skill applies to a step below

Do not skip this step. The documents contain decisions (e.g. which SQLite driver, which logger, exact interface signatures) that must match what you scaffold. Verify against them as you go.

## 6. Go Source Stubs

For every `.go` file listed in §1, create a stub. A stub contains the `package` declaration, the file's purpose in a comment, and empty skeletons of all exported types, interfaces, and functions it will eventually hold — but no implementation logic yet. This ensures `go build ./...` passes from day one.

### Rules for writing stubs

- Match the package name to the directory name.
- Declare all exported types and interfaces exactly as specified in `.agent/FUNCTIONAL_SPEC.md` and `.agent/AI_AGENT_GUIDE.md`.
- Functions return zero values and `nil` errors, or `errors.New("not implemented")` where the return type requires a non-nil error to be meaningful.
- Do not use `panic("not implemented")` — the binary must be runnable.
- Add a `// TODO(M{N}-{ID}): implement` comment on every stub body referencing the task from `TASKS.md` that will fill it in.

### Critical stubs to get right

The following files define the interfaces everything else depends on. Their signatures must be identical to the spec. Cross-check each one against the document cited.

## 7. Stub Out Remaining Command Files

For each file in `cmd/` (other than `root.go`), create a stub that:

1. Declares the top-level `cobra.Command` variable with `Use`, `Short`, and `Long` fields populated from `FUNCTIONAL_SPEC.md §2`.
2. Declares all sub-command variables with the same fields.
3. Has an `init()` that calls `rootCmd.AddCommand(...)` and sub-command wiring.
4. Has a `RunE` stub that returns `errors.New("not implemented")`.

**Do not leave `RunE` as `nil`** — nil RunE causes Cobra to print usage instead of an error, which makes it hard to tell that a command is unimplemented.

---

## 8. Stub Out Internal Packages

For each `internal/` package, the stub must:

- Declare the package.
- Import only what is needed for the interface/type declarations (no unused imports — `go build` will fail).
- Contain empty function bodies with `// TODO` comments.

The one exception is `internal/apperrors/errors.go` — this is fully implemented in step 6 and needs no further work.

## 9. Verify the Scaffold

Once all files are created, run the following commands. All must succeed before this task is complete.

```bash
# Tidy dependencies
go mod tidy

# Check it compiles
go build ./...

# Check tests (all stubs pass vacuously)
go test ./...

# Check for lint issues introduced by the scaffold
golangci-lint run ./...
```

If any command fails, fix the error before proceeding. Do not leave a broken scaffold.

**Common scaffold errors to watch for:**

| Error | Likely cause |
|---|---|
| `imported and not used` | Stub imports a package it doesn't use yet — remove the import |
| `undefined: models.SomeType` | Type missing from `pkg/models/models.go` — check spec §1 |
| `cannot use *X as type Y` | Interface not implemented — add missing method stubs |
| `declared but not used` | Variable declared in stub but never referenced — use `_` or remove |
